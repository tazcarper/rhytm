Building a Feature

Build a feature by just asking

A real, start-to-finish example of a non-technical person adding a feature to the site with Claude — not a layout tweak, but a new admin screen backed by the database. You describe what you want to be able to do; Claude builds it and packages it for your developer to review.

1What this example shows

Most of the changes you make are presentation — wording, spacing, colors, moving things around. This example is bigger: a feature. It answers one question:

What does it look like when you ask Claude for something you want to control yourself — something that needs a new admin screen and a place in the database — and turn it into a reviewable change?

The example feature: an admin page where staff can edit the homepage hero banner — the big welcome block at the top of the public site. The label, the headline, the supporting text, the two buttons, and an optional background image — all editable, with the homepage showing whatever you set. No developer needed every time the wording changes.

First, though — is it already editable in the dashboard? Before asking Claude to change anything, check the admin area. A lot is already editable there — property info, pricing, services & add-ons, adventures, FAQ & gear, waiver wording, and more. If what you want to change is one of those, just change it in /admin — it's live and instant, with no code change, no review, and no developer. You only need this “build a feature” path when the thing is hardcoded and you want to be able to change it yourself going forward — like the banner below.

2Which path is my change?

Three kinds of change, three different paths. Figuring out which one you're in saves everyone time.

If your change is…Do thisWho's involved
Already editable in the dashboard (pricing, property info, adventures, FAQ, waiver text…)Just edit it in /adminOnly you — live & instant
Hardcoded, but you'll change it again and again (like this banner)Build a feature so it becomes dashboard-editable (this guide)You + Claude build it; developer reviews once
Hardcoded and genuinely one-off (a layout tweak, a fixed bit of copy)A small change Claude makes directlyYou + Claude; developer reviews

The trap to avoid: using the third row for something that's really the second. If you'll keep tweaking it, don't hardcode it — graduate it into the dashboard once, and you'll never need a developer for it again.

3How you'd actually ask for it

You don’t need the words “database,” “migration,” or “security rule.” You describe what you want to be able to do. Here’s the whole conversation, paste by paste.

The opening ask

“On the homepage there’s a big welcome banner at the top — the ‘Your day in the Texas Hill Country starts here’ part. Right now I have to ask a developer every time I want to change that wording. I’d like to edit it myself. Can you add a page in the admin area where I can change the little label, the big headline, the paragraph under it, and the two buttons? And let me put a background image behind it too.”

That single message is enough. Claude creates a place to store the banner, builds the admin page with a form, makes the homepage read from it, and tells you the one part your developer will need to apply.

Seeing it — look in your own browser

You don’t ask Claude to “show you” the page — you look at it yourself in your web browser. Claude keeps the local copy of the site running at http://localhost:3000; open that and refresh to see every change as you go.

  • The banner → visit http://localhost:3000
  • The editor → sign in, then visit http://localhost:3000/admin/homepage

If the site isn’t running (the page won’t load), just say so and Claude will start it for you:

“The site isn’t loading — can you make sure it’s running so I can look at it in my browser?”

Iterating on the look

“The headline is too big — can you make it a bit smaller?”

“Put more space between the two buttons.”

“When I add a background image the text is hard to read — can you darken the image a little so white text shows up?”

Changing the content (the everyday use, once it’s live)

“Change the headline to ‘Your Hill Country weekend starts here.’”

“Make the first button say ‘Book a visit’ and point it at the booking page.”

“Hide the second button for now.”

When you’re happy

“This looks great. I’m done — open the pull request so my developer can review it.”

If you get stuck

“Claude says something is blocked — what does that mean?”

That’s the safety guardrail: the action belongs to your developer (anything touching the real live website or its data). Claude has already captured it in your pull request. Just keep going.

4What you're really asking for

The point of the example: plain-language requests map cleanly onto real engineering pieces. You only ever write the left column. Claude produces the right column.

What you sayWhat Claude builds
“store the banner so I can edit it”a place in the database for the banner (a “migration”)
“only staff should be able to change it”a security rule: the public can view it, only admins can edit it
“a page in the admin area to edit it”a new /admin/homepage screen with a form
“the homepage should show what I set”the homepage reads the saved values and displays them
“let me put an image behind it”an image field, plus a subtle darkening so the text stays readable

5Who does what

Your part

  • Describe what you want to be able to do
  • Look at it on your own computer and say whether it’s right
  • Iterate on wording and look with Claude
  • Say “I’m done” so Claude opens the pull request

Your developer's part

  • Review the change
  • Apply the one “database change” to the live site
  • Publish it

One thing to expect: because this feature includes a database change, you’ll see it working right away on your own computer (localhost:3000), but the shared preview link won’t show it correctly until your developer applies that change. Trust your local view — that’s the source of truth while you work.

6Under the hood (for your developer)

For the curious — and for the developer reviewing it — the feature is a small, standard full-stack slice that follows the project’s existing patterns (the same “config in the database” approach as the per-property taglines). It adds:

  • a migration creating the homepage_hero row, with security rules (public read, admin-only write), seeded with the current wording;
  • a small read service and an admin write service (with input validation);
  • a thin admin page + edit form + a “Homepage” nav link;
  • the homepage reading those values instead of fixed text.

The finished code is kept as a permanent example on its own branch (deliberately not merged), so anyone can study the exact diff:

git diff main..feature/editable-homepage-hero

The developer’s review checklist — apply the migration locally, test the admin edit, hand-test the security rule (a non-admin must not be able to write), then apply to production and publish — lives with the code.

The full write-up (every file, the prompt library, and the developer review runbook) lives in docs/examples/editable-homepage-hero.md in the repo. New to making changes at all? Start with the Contributor setup guide.
